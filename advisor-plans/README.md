### Considered and REJECTED (vetted against TR-1 / one-WAL / scope — do not re-propose)
- **GPU batched radix top-k** (RadiK/AIR-Top-K) for the final k-selection — materialize-all + blocking,
  **violates TR-1**. GPU's defensible use is the offline build (plan 008) and the per-candidate distance
  kernel, never the rank merge.
- **GraphBLAS masked SpMV / Gunrock bulk-synchronous frontier** for multi-hop expansion — level-synchronous,
  materializes the whole frontier, **violates TR-1** on the emission path. (The membership mask *is*
  conceptually a GraphBLAS mask — but as documentation, not an executor.)
- **Sirius / TQP GPU columnar relational engine** (libcudf) for the relational leg — bulk/vectorized,
  blocking; **violates TR-1** as a canonical executor.
- **DuckPGQ on-the-fly CSR build** over the reachable set — blocking materialization **and** against
  golden rule 3 (TriDB's thesis is a *native persistent* adjacency AM). It's the anti-pattern that
  confirms the native-AM choice.
- **HippoRAG as published** (PPR to convergence, then sort all passages) — materialize-all + sort-before-emit,
  the exact blocking shape ADR-0012 must avoid. The compliant form (bounded forward-push PPR) is plan 007.
- **DuckDB push-based executor rewrite** — out of scope (XL rewrite of a PG-13.4 pull-based fork for no
  required gain); the salvageable streaming top-k micro-pipeline is already implicit in `tjs_open`.
- **Custom WAL Resource Manager (`RegisterCustomRmgr`)** for the graph AM — **PG15+, unavailable on the
  PG-13.4 fork**. GenericXLog is the *correct* one-WAL choice on 13.4 (gives SM-5 free), not a limitation.
  Revisit only after a rebase past PG15 *and* if logical decoding of graph edges is ever required.
- **NaviX CSR-HNSW substrate unification / full Filtered-DiskANN StitchedVamana / WCOJ leapfrog-triejoin**
  — deferred (research-only) for the single-linear-hop v1 canonical query; the borrowable kernels
  (NaviX in-loop selectivity → #7; filtered entry-point selection → #7) are taken without the heavy parts.

### Execution + review outcome — ROUND 2, real validation (2026-06-29)
A second execution round ran each plan to its *real* completion on this box, which turned out far more
capable than the conservative "GX10-gated" labels implied: the `tridb/msvbase:dev` **x86 engine image is
present** (and PGXS-builds `src/` from the worktree at test time), the **HotpotQA + SIFT datasets are
local**, and the **GX10 (`ssh spark`) is reachable**. Each was executed by an isolated-worktree executor
and reviewed (advisor re-ran the checks, read every diff, audited the new tests — no self-report trusted).
Verdict: **all four APPROVED with real, measured results.**

- **006 — DONE, engine-validated on the x86 image (not just authored).** The metapage C **compiled and
  passed the full native-AM engine suite** on `tridb/msvbase:dev`: `graph_am_test` PASS, `join_order_test`
  FR-6 parity PASS, `graph_edge_count_test` PASS (init 0 → branches 1+2 = 4 → multi-page chain = 1504),
  `crash_recovery` **both scenarios PASS** — including scenario 2 proving an *uncommitted* edge's increment
  is absent after a crash (`gm_edge_count = 0`), the abort-safety the plan required. Step 5 Makefile wiring
  applied and built. `make test` 152 / lint clean. Frozen `join_order.c` untouched (verified empty diff).
  **Executor caught + corrected a real bug in the *authored test*:** it had asserted a clean `ROLLBACK`
  reverts the counter (1506→1504), which is impossible — GenericXLog gives crash-recovery REDO, not
  in-process UNDO — and fixed it to the honest v1 behavior (stays 1506; the raw counter is not in-process
  abort-aware, like `gph_vertex_count`), with crash-recovery scenario 2 proving the abort-safety that *is*
  guaranteed. Two harness files were edited (in-scope) so the engine build resolves the new `gph_page.h`
  include. Branch `advisor/006-metapage-degree-stats` (`…ad05217`, `4ccc8ff`). **Remaining:** only an
  optional ARM sign-off of the identical C on the GX10 (no ARM-specific code) — not a correctness gate.
  *Nit:* the `graph_edge_count_test.sql` header comment (lines 4-5) still says "rolls back atomically on
  ABORT" and should be reconciled with the corrected body.
- **007 — DONE, real HotpotQA, and it BEATS the blocking oracle.** Ran on the real 1490-para / 745-edge /
  150-question manifest (BGE-768): **FR-fused recall@5 0.937 and RRF-fused 0.907 both exceed the (A)
  blocking oracle's 0.883**; FR recall@10 **0.987** > vector-only 0.967 — all while examining **~0.71% of
  the corpus** (the TR-1 early-termination proxy, with huge margin). The streaming TR-1-pure composition is
  not merely within tolerance of the blocking reference — it surpasses it. Honest caveats logged in the
  ADR-0012 addendum: `ppr_only` alone (0.863@5) underperforms vector — graph is a *fusion* signal, not a
  standalone retriever on this sparse graph; and `r_max` is inert here (flat curve, mean reach ≈ 10 nodes),
  so its recall/examined tradeoff only bites on a denser/at-scale graph (GX10 follow-up). This de-risks the
  committed `tjs_open` (B) build with real evidence. Branch `advisor/007-tjs-open-ranking` (`b8bf340`).
- **008 — DONE (host), real SIFT; CAGRA build+export now VALIDATED on the GB10.** RaBitQ on real
  `sift-128` (two agreeing runs): **4-bit + in-scan rerank → recall@10 = 1.0 at 7.5×** compression; 8-bit
  raw 0.99. This *corrected* the synthetic finding — **1-bit is dead on real data** (rerank doesn't rescue
  it: 0.07@R=100; the synthetic 1-bit viability was an easy-clustering artifact), so the honest
  recall-preserving lever is ~7.5× (4-bit), not 8-32×. The design note's footprint/toggle analysis stands
  (zero serving-path GPU footprint, default-OFF `WITH_CUVS`). **CAGRA UNBLOCKED (2026-06-29):** cuVS
  **26.06** was provisioned on the GB10 (aarch64 + CUDA 13 + sm_121, isolated `~/cuvs-env`), and the build
  path is **verified live** — `cagra.build` (20k×128 in ~1.96 s) + `hnsw.from_cagra(IndexParams(), idx)` →
  `hnsw.save` produces a 13.2 MB HNSW file. `scripts/gpu_build_index.py` was reconciled to the real cuVS
  26.06 API (the authored `from_cagra(index)` form was wrong). **Remaining:** the recall A/B (load the
  exported file through the fork's `hnsw` AM vs a CPU-built index) — now a staging+format-check task, not a
  provisioning blocker. Branch `advisor/008-gpu-build-quant` (`e0641f4`) + the 2026-06-29 doc/script updates.
- **009 — DESIGN-DONE; prototype built + benchmarked on x86 → lean NO-GO (the right call).** Greenlit, the
  delta-tail CSR-lite prototype was built and run on the `tridb/msvbase:dev` image: it **compiles, is
  correct (output now globally sorted, `nondecreasing=t`), and is FR-7-safe — zero divergence on both
  txn-atomicity (200 iters) and crash-recovery** (a real result: the re-layout does not break MVCC/crash
  atomicity). Bulk-load write cost is within noise (−2.5%/−0.3%). **But the verdict is lean NO-GO** because
  the traversal page-read win — the entire justification — **could not be evaluated on x86**: the spike kept
  the `gph_next_pageno` chain (no physical contiguity to measure) and a warm 16 MB buffer pool can't
  exercise 128 GB residency pressure. Two genuine new findings landed in the design note: **GenericXLog caps
  at 4 buffers/record**, so an atomic hub re-sort needs a shadow-extent swap (a custom rmgr is golden-rule-2
  forbidden); and `gph_neighbors` re-reads a page per `Next()` (no-pin contract), distorting the metric. A
  contiguity-delivering prototype + pinned scan on the **GX10** is the GO precondition; compression (#14) and
  WCOJ stay deferred. **The prototype is NOT merged** (NO-GO) — it lives on `spike/009-csr-lite-prototype`
  (`c90683c`) for the GX10 re-run; only the design-note verdict (§7 numbers + §8) landed on master.

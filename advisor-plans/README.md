### Considered and REJECTED (vetted against TR-1 / one-WAL / scope — do not re-propose)
- **GPU batched radix top-k** (RadiK/AIR-Top-K) for the final k-selection — materialize-all + blocking,
  **violates TR-1**. GPU's defensible use is the offline build (plan 008) and the per-candidate distance
  kernel, never the rank merge.
- **GraphBLAS masked SpMV / Gunrock bulk-synchronous frontier** for multi-hop expansion — level-synchronous,
  materializes the whole frontier, **violates TR-1** on the emission path. (The membership mask *is*
  conceptually a GraphBLAS mask — but as documentation, not an executor.)
- **Sirius / TQP GPU columnar relational engine** (libcudf) for the relational leg — bulk/vectorized,
  blocking; **violates TR-1** as a canonical executor.
- **DuckPGQ on-the-fly CSR build** over the reachable set — blocking materialization **and** against
  golden rule 3 (TriDB's thesis is a *native persistent* adjacency AM). It's the anti-pattern that
  confirms the native-AM choice.
- **HippoRAG as published** (PPR to convergence, then sort all passages) — materialize-all + sort-before-emit,
  the exact blocking shape ADR-0012 must avoid. The compliant form (bounded forward-push PPR) is plan 007.
- **DuckDB push-based executor rewrite** — out of scope (XL rewrite of a PG-13.4 pull-based fork for no
  required gain); the salvageable streaming top-k micro-pipeline is already implicit in `tjs_open`.
- **Custom WAL Resource Manager (`RegisterCustomRmgr`)** for the graph AM — **PG15+, unavailable on the
  PG-13.4 fork**. GenericXLog is the *correct* one-WAL choice on 13.4 (gives SM-5 free), not a limitation.
  Revisit only after a rebase past PG15 *and* if logical decoding of graph edges is ever required.
- **NaviX CSR-HNSW substrate unification / full Filtered-DiskANN StitchedVamana / WCOJ leapfrog-triejoin**
  — deferred (research-only) for the single-linear-hop v1 canonical query; the borrowable kernels
  (NaviX in-loop selectivity → #7; filtered entry-point selection → #7) are taken without the heavy parts.

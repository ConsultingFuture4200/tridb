# TriDB fused (graph+vector) link prediction vs the cosine-only lower bound (v0.1.0)

**What this is.** The cheapest direct test of TriDB's core thesis on the real full-Wikipedia
graph (DEV-1354): given a candidate pool generated by embedding similarity, does **graph
topology re-rank the real hyperlinks higher than cosine alone?** If yes, topology adds
predictive signal beyond the embedding; if fused ≈ cosine, the thesis gets no support here.
**This is a predictive-signal test, NOT a latency claim** (ADR-0017: the I/O-bound regime is
expected-fail; no speed win is claimed).

TL;DR
- **Topology adds a real but modest signal.** On the identical 50-candidate cosine pool,
  Adamic-Adar reranking lifts overlap@10 from the **0.110 cosine lower bound to 0.133**
  (+21% relative), and RRF-fused cosine+AA reaches 0.130 (+18%). Both clear the cosine
  ranker; neither is a landslide.
- **Honest nuance:** Adamic-Adar *alone* (0.133) slightly **beats** the RRF fusion (0.130) —
  within a semantically-close pool the topological prior is the stronger ranker, and blending
  it back with cosine dilutes it a hair. The graph is the load-bearing signal here, not the
  embedding.
- **Sanity check holds:** cosine-only reproduces the baseline exactly (0.110 vs 0.1101),
  confirming this is an apples-to-apples rerank of the same pool.
- **node2vec: skipped by design** (training-based, multi-hour on 6.9M nodes / 224M edges;
  out of scope for the cheapest-test remit). CN/AA are training-free and answer the question.
- **Scale is real, not a slice:** 6,900,039 embeddings, one streaming pass over the full
  224,475,283-edge graph (221.1M valid after endpoint-membership filter), on the Spark GB10.

## Method

**One canonical protocol, four columns.** Candidate generation is bounded to each source's
cosine **top-50** neighbours from the cuVS CAGRA index (the same GPU path as the baseline).
Every column then scores overlap@10 = *fraction of the top-10 reranked candidates that are
already an out-edge of the source* — the same metric as the 0.110 lower bound. Because the
pool and metric are identical for all four columns, **cosine-only reproduces 0.110 as a
built-in sanity check** and the other three are pure reranks of the same pool. This respects
TR-1 / early termination: topology never scores all O(N²) pairs, it only reorders a bounded
pool, and the adjacency materialized is bounded to the ~246k source+candidate nodes, not the
full 6.9M.

**Topological scorers** (no training, computed from adjacency), predicting a directed link A→B:

| Scorer | Definition | Notes |
|---|---|---|
| Common Neighbours | `CN(A,B) = |succ(A) ∩ succ(B)|` | successor/**bibliographic-coupling** variant |
| Adamic-Adar | `AA(A,B) = Σ_{z ∈ succ(A)∩succ(B)} 1/log(indeg z)` | rare shared targets weigh more |

We use **successor sets** (out-neighbours) on both endpoints. This is deliberate: it is
bounded by out-degree (mean 31.5), so intersections stay cheap on 224M edges. The undirected
variant would pull in million-degree hub *in*-neighbourhoods and blow the compute budget.
Neither score reads the A→B bit, so there is **no label leakage** into the features — but it
*is* a genuine in-family prior (articles that link to the same targets tend to link to each
other), which is exactly the topological signal under test.

**Fusion:** reciprocal-rank fusion (RRF, `rrf_k=60`) of the cosine rank and the Adamic-Adar
rank within the pool: `score(B) = 1/(60+rank_cos) + 1/(60+rank_aa)`.

**node2vec (skipped).** Training node2vec on 6.9M nodes / 224M edges means a multi-billion-token
random-walk corpus plus a skip-gram training pass — a multi-hour-to-day job needing a training
pipeline we do not have here, and out of scope for the cheapest-test remit. It is reported as
`node2vec_included=false` in the results JSON, not silently omitted.

**Honest boundaries.** These embeddings are **dim-384 float32** (bge-small), **NOT** the spec's
dim-768 float8 I/O-pressure basis — this is the compute-bound-ish regime, not the I/O regime, so
no latency conclusion is drawn. Counts are reconciled to the real corpus: **6,900,039 articles /
224,475,283 edges** (the extractor manifest bug lost shards 28/49/71; we trust the rebuilt shard
files — 221,135,287 edges survive the both-endpoints-are-articles filter, ~1.5% dropped).

## Results

Real enwiki, 5000 sampled sources (seed 42), cosine top-50 pool, overlap@10, on the Spark GB10.
Mean out-degree 31.5. **Pool base rate 0.042** — only ~2.1 of the 50 pooled candidates are true
out-edges on average, so a random rerank scores ~0.042 and the perfect-rerank ceiling is
~0.21 (2.1/10). All rankers are measured against that floor and ceiling.

| Ranker | overlap@10 | vs cosine | vs random-in-pool (0.042) |
|---|---:|---:|---:|
| **cosine-only** (= baseline lower bound) | **0.110** | — | 2.6× |
| common-neighbor only | 0.128 | +0.018 (+16%) | 3.0× |
| **adamic-adar only** | **0.133** | **+0.023 (+21%)** | **3.1×** |
| fused (RRF cosine+AA) | 0.130 | +0.020 (+18%) | 3.1× |

Sanity: cosine-only 0.110 vs the standalone baseline 0.1101 — identical, as designed.

Run cost (GB10): CAGRA build over 6.9M vectors 49.5s; one streaming pass over all 72 edge
shards (224.5M rows) 28.9s; scoring 5000×50 intersections negligible. End-to-end a few
minutes — cheap, exactly as intended.

## Verdict

**Topology adds signal — modestly. (yes, small)**

Graph structure concentrates the true hyperlinks measurably higher than embedding proximity
alone: Adamic-Adar lifts overlap@10 by +0.023 absolute (**+21% relative**) over the 0.110
cosine lower bound, on the real 224M-edge graph, with candidate generation held identical. The
lift is real and reproducible, not noise — but it is a nudge, not a step-change: absolute
overlap moves from 0.110 to 0.133 against a 0.21 ceiling.

The sharpest honest finding is that **Adamic-Adar alone (0.133) beats the RRF fusion (0.130)**.
Within a pool that is already semantically close, the topological prior is the *stronger* single
ranker, and fusing it back with cosine slightly dilutes it. So the graph is not merely additive
decoration on top of the embedding here — on this pool it is the better ranker of the two. That
supports the tri-modal thesis (topology carries predictive signal the vector index does not) while
refusing to oversell it: the effect is small in absolute terms because most of a Wikipedia
article's global nearest embedding-neighbours are simply unlinked topical peers.

This is a bounded-pool reranking result on dim-384 f32 embeddings. It says nothing about latency
(ADR-0017 I/O regime is expected-fail and untested here), and it is a lower bound on what the
production `tjs_open` join — which can traverse multi-hop topology rather than only 1-hop
successor overlap — could extract.

## Reproduce

```bash
# On the Spark GB10 (GPU CAGRA + 224M-edge pass), from the repo root:
.venv/bin/python tools/wiki_linkpredict_fused.py \
    --corpus data/wiki/enwiki --emb-in data/wiki/enwiki/emb \
    --sample 5000 --seed 42 --pool 50 --k 10 \
    --json-out bench/out/wiki_linkpred_fused.json
```

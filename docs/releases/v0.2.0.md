# TriDB v0.2.0 — release notes

**TL;DR** — TriDB now installs on **stock PostgreSQL 16/17 + pgvector** as two plain
extensions (no forked Postgres), runs vector search, graph traversal, and relational
filtering in **one query plan under one transaction manager**, and ships with: a
bounded (early-terminating) graph leg, a measured better-than-baseline default for
seedless graph scoring (Personalized PageRank grading), an MCP server that turns any
MCP-capable agent into a TriDB user, a logical backup/restore procedure that actually
carries the graph, in-place `ALTER EXTENSION` upgrades from 0.1.0, and an enforced
single-writer contract. Every number below links to the in-repo document that
measured it.

Extension versions in this release: `graph_store_am 0.2.0`, `tjs_pg 0.2.0`
(both `.control` files; the release image and the `v0.2.0` tag carry the same stamp).

## What TriDB is

One PostgreSQL process holding three coordinated stores — pgvector for similarity,
a native adjacency-list graph access method for traversal, and ordinary relational
tables for filtering — fused by a single operator (`tjs_open`) that keeps one global
top-k during execution, so intermediate results never materialize in full. The
alternative it is measured against is the common bolt-on stack (a vector DB + a
graph DB + a relational DB glued together in application code), which pays
cross-system round-trips and cannot make a cross-store transactional guarantee.

## Try it

```bash
docker build -f scripts/pg17/Dockerfile.release -t tridb/postgres-trimodal:pg17 .
docker run -d --name trimodal -e POSTGRES_PASSWORD=secret tridb/postgres-trimodal:pg17
docker exec -it trimodal psql -U postgres \
  -c 'CREATE EXTENSION vector;' \
  -c 'CREATE EXTENSION graph_store_am;' \
  -c 'CREATE EXTENSION tjs_pg;'
```

Full install guide (Docker, from-source PGXS, upgrade, backup, concurrency):
[`docs/INSTALL_stock_pg.md`](../INSTALL_stock_pg.md).

## Highlights

### 1. Runs on stock PostgreSQL — and the fusion win survived the un-fork

TriDB began as a fork of Microsoft's MSVBASE (PostgreSQL 13.4). This release's engine
path needs no fork: `graph_store_am` (the native graph store) and `tjs_pg` (the fused
operator) build as standard PGXS extensions on stock PostgreSQL 16 and 17 with
pgvector, exercised by an always-on CI matrix on both majors.

The reason to fuse at all — avoiding the materialize/transfer/prune cycle of a
multi-system stack — was re-measured after the un-fork on a pinned 1,002,331-entity
Wikidata slice, same queries, same graded recall, against a Milvus + Neo4j + Postgres
baseline ([`docs/gate_b_spike_v0.1.0.md`](../gate_b_spike_v0.1.0.md)):

| Platform | TriDB (fused filter-first) | multi-store baseline | speedup |
|---|---|---|---|
| PostgreSQL 13.4 fork, 32KB pages | 0.27 ms @ recall 0.992 | 3.16 ms @ 0.986 | 11.90× |
| **Stock PostgreSQL 17 + pgvector, 8KB pages** | **0.14 ms @ recall 0.992** | 3.34 ms @ 0.986 | **23.68×** |

On a real 200k-article Wikipedia hyperlink corpus, the in-process fused operator beat
an app-side Milvus→Neo4j→pgvector pipeline at matched recall@10 by 11.5× (1-hop) /
3.26× (2-hop) over loopback, and 16.7× / 10.6× when the baseline pays real network
round-trips ([`docs/benchmark_wiki_fusion_v0.1.0.md`](../benchmark_wiki_fusion_v0.1.0.md)).
The win is eliminating cross-system round-trips, not faster ANN.

The other half of the value is transactional: with all three stores under one WAL,
injected failures left TriDB with 0 torn multi-store writes where the bolt-on stack
had 42/42 torn; crash recovery rolled uncommitted work back atomically across all
three legs; concurrent readers saw 1.0% torn cross-store states vs the baseline's
76.7% ([`docs/benchmark_wiki_consistency_v0.1.0.md`](../benchmark_wiki_consistency_v0.1.0.md);
see honest limits below for the residual 1.0%).

### 2. Bounded graph leg — no blocking operators (ADR-0020)

The stock operator's graph leg previously materialized a full bounded-depth BFS
before serving its first row. It now runs a bounded pull-based iterator under an
explicit work budget (`tjs.graph_work_budget`), preserving early termination
end-to-end, with a per-query censor flag (`tjs_open_graph_censored()`) that reports
whenever the budget truncated the graph work — so a benchmark headline can never
silently hide a truncated traversal.
Decision record: [`docs/decisions/0020-stock-tjs-incremental-graph-leg.md`](../decisions/0020-stock-tjs-incremental-graph-leg.md).

### 3. PPR-graded scoring is now the default for seedless retrieval (ADR-0021)

When a query gives no anchor entity ("seedless": rank everything, let the graph
reinforce), graph-sourced candidates are now scored by a bounded forward-push
Personalized PageRank pass instead of binary reachability. The flip was gated on two
recall measurements with independent gold, and PPR won every matched point on both
([`docs/decisions/0021-ppr-default-graph-scoring.md`](../decisions/0021-ppr-default-graph-scoring.md)):

| Gate | Corpus | Gold | PPR vs membership |
|---|---|---|---|
| Multi-hop QA | HotpotQA (1,490 nodes) | task gold (supporting paragraphs) | recall@5 **0.927 vs 0.880** (+4.7 pt), joint@5 +9.3 pt; wins all 12 points |
| Held-out link prediction | English Wikipedia, 200k articles, 14.68M real hyperlink edges | held-out editorial hyperlinks | recall@20 **0.119–0.123 vs 0.081–0.083** (+47% relative); wins all 18 matched points |

The old behavior stays one `SET tjs.graph_scoring = 'membership'` away, anchored
(filter-first) queries are unchanged, and the measured budget guidance — the budget
is a latency knob, not a recall knob, on hyperlink-density graphs — is documented in
[`docs/INSTALL_stock_pg.md`](../INSTALL_stock_pg.md) ("Seedless graph scoring").

Separately, on the reproducible HotpotQA harness (pinned data and seeds, graded
against gold, runs on a plain x86 box), injecting real graph bridges lifts multi-hop
joint evidence recall@5 by +15.6 pt over vector-only retrieval
([`docs/benchmark_public_repro_v0.1.0.md`](../benchmark_public_repro_v0.1.0.md)).

### 4. MCP agent-memory server

`tools/tridb_mcp.py` exposes the release image as agent memory over the Model
Context Protocol: five stdio tools (`store_memory`, `connect`, `recall`,
`neighbors`, `memory_stats`). A stored memory is a relational row + a vector + typed
graph links, inserted in one transaction; `recall` runs the fused operator under the
PPR default and passes the engine's honesty probes (censor flag, termination reason)
through to the caller. `make mcp-demo` is the one-command proof.
Docs: [`docs/mcp_agent_memory_v0.1.0.md`](../mcp_agent_memory_v0.1.0.md).

### 5. Logical backup/restore that carries the graph

Before this release, a `pg_dump`/restore cycle **silently lost the entire graph leg**
(exit code 0, no warning — the restored database looked healthy). Physical backups
were always safe (one WAL), but logical dumps dropped the custom-format graph pages.
Now the extension config tables ride `pg_dump` natively and two functions
(`gph_dump_vertices()` / `gph_dump_edges()`) export the topology for replay; the
documented procedure round-trips byte-identically and is gated by
`scripts/graph_dump_restore_test.sh` (including a corrupted-dump negative control).
Procedure: [`docs/INSTALL_stock_pg.md`](../INSTALL_stock_pg.md) ("Backup and restore").

### 6. In-place upgrades and an enforced single-writer contract

- **Upgrades**: both extensions ship versioned upgrade scripts; an existing 0.1.0
  install upgrades in place with `ALTER EXTENSION ... UPDATE TO '0.2.0'` after
  installing the new binaries. The gate (`scripts/extension_upgrade_test.sh`)
  installs a genuine vendored 0.1.0, loads a tri-modal corpus, upgrades, and asserts
  a byte-identical probe. From 0.2.0 on, released surface changes always ship as
  upgrade scripts (see `CONTRIBUTING.md`).
- **Single writer, now enforced**: structural graph writes take a transaction-scoped
  exclusive advisory lock per graph. A second concurrent writer blocks (does not
  error) until the first commits or aborts; readers never take the lock and stay
  MVCC-consistent. Gate: `scripts/graph_writer_lock_test.sh`.
  Details: [`docs/INSTALL_stock_pg.md`](../INSTALL_stock_pg.md) ("Concurrency").

## Honest limits — read before benchmarking

- **Single writer.** Concurrent structural writers serialize (by design, enforced);
  multi-writer snapshot isolation for the graph store is deferred work. Readers are
  concurrent and MVCC-filtered.
- **Graph-leg read visibility.** In the live consistency measurement, TriDB's
  residual 1.0% torn reads come from the native graph leg being commit-visible
  rather than snapshot-isolated (the vector + relational legs measured 0.0%)
  ([`docs/benchmark_wiki_consistency_v0.1.0.md`](../benchmark_wiki_consistency_v0.1.0.md)).
- **What was measured at what scale.** Filter-first fusion: 1M entities (Wikidata,
  stock PG17 and the fork — [`docs/gate_b_spike_v0.1.0.md`](../gate_b_spike_v0.1.0.md)).
  Seedless fusion speed + consistency: 200k articles / 14.68M edges (English
  Wikipedia). PPR QA gate: a 1,490-node HotpotQA corpus. The 1M seedless
  head-to-head is blocked on a non-reproducible HNSW build failure in the *fork*
  vector leg and was not run — no 1M seedless latency is claimed anywhere
  ([`docs/benchmark_wiki_fusion_v0.1.0.md`](../benchmark_wiki_fusion_v0.1.0.md)).
- **Fork-hardware-gated claims.** Numbers measured on the MSVBASE PostgreSQL 13.4
  fork (the original reference vehicle) require its target hardware (NVIDIA GX10 /
  DGX Spark, ARM64+CUDA, 128 GB). The 128 GB headline benchmark **has not been
  run**. The recall/effort trade-off of the fork's seedless retrieval is a curve,
  not a point — at 100k scale it spans 58.5% exact-parity (fast default, 3.6% of
  corpus examined) to 100% (higher termination threshold, 20.1% examined); never
  quote one end without the knob setting (curve table:
  [`docs/gtm_opensource_v0.1.0.md`](../gtm_opensource_v0.1.0.md), risk R1; also
  summarized in the README's SM-4 note).
- **One target the design missed.** The intermediate-result-reduction metric (SM-1,
  target ≥5× vs baseline) measured 1.07× at stand-in scale — a FAIL, recorded, not
  restored by later work ([`docs/benchmark_results_v0.1.0.md`](../benchmark_results_v0.1.0.md)).
- **`pg_dump` alone is not a full logical backup.** Use the plan-099 procedure above
  for the graph topology.
- **Query surface is deliberately narrow.** One canonical SQL/PGQ + pgvector query
  template for v1 (`graph_store.graph_query($$...$$)`); off-template text is
  rejected. No new query language.

## Compatibility

- Stock PostgreSQL 16 and 17 (x86_64 in CI on every push; aarch64 validated
  out-of-band on a DGX Spark).
- pgvector required (`tjs_pg` declares `requires = 'vector, graph_store_am'`).
- Upgrading from 0.1.0: install the new `.so` files first, then
  `ALTER EXTENSION graph_store_am UPDATE TO '0.2.0'; ALTER EXTENSION tjs_pg UPDATE TO '0.2.0';`
  Data written under 0.1.0 is untouched by the upgrade.
- Behavior change: seedless (no-anchor) queries now default to PPR-graded scoring
  (ADR-0021). `SET tjs.graph_scoring = 'membership'` restores the 0.1.0 ranking
  byte-identically. Anchored/filter-first queries are unchanged.

## Provenance

TriDB is a clean-room realization of AkasicDB (SIGMOD Companion '26), built from
MSVBASE (VBASE, OSDI '23), extending Chimera's (PVLDB 18(2)) dual-store design to a
triple store. License: MIT ([`LICENSE`](../../LICENSE)), consistent with the
upstream `microsoft/MSVBASE` base.
